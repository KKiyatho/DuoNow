## 5. 커밋 & 코드 리뷰 규칙

### Git 브랜치 전략
```
main (프로덕션 배포용)
├─ staging (통합 테스트용)
│  ├─ feature/auth
│  ├─ feature/matching
│  ├─ feature/chat
│  └─ feature/review
└─ develop (개발용)
   ├─ feature/{name}
   └─ bugfix/{issue}
```

### 커밋 메시지 규칙
```
[Type] Brief description

feat: 새 기능
fix: 버그 수정
refactor: 코드 정리
test: 테스트 추가
docs: 문서 수정
chore: 환경설정 변경

예시:
[feat] Add matching algorithm with 4-step fallback
[fix] Fix WebSocket reconnection timeout
[test] Add e2e test for login flow
```

### 코드 리뷰 체크리스트
- [ ] 코드 컨벤션 준수
- [ ] 주석/문서 충분
- [ ] 테스트 커버리지 > 80%
- [ ] 성능 영향도 검토
- [ ] 보안 이슈 없음

## 7. 위험 관리 계획

### Risk 1: 매칭 알고리즘 성능 미달
**발생 확률:** 중간 (40%)  
**영향도:** 높음 (데모 실패 가능)

**대응:**
- Week 3 월요일부터 성능 테스트 시작
- 큐 최적화 (인덱싱, 캐시)
- 폴백 매칭 옵션 사전 준비
- 대체 알고리즘 고려 (Redis Sorted Set)

---

### Risk 2: 실시간 채팅 지연
**발생 확률:** 낮음 (20%)  
**영향도:** 중간 (사용자 경험 저하)

**대응:**
- Supabase Realtime vs Firebase vs Socket.io 비교 평가 (Week 1)
- 메시지 배칭 (10초마다 묶음 전송)
- 오프라인 메시지 큐 준비
- 재연결 로직 강화

---

### Risk 3: 발표 직전 서버 장애
**발생 확률:** 낮음 (15%)  
**영향도:** 매우 높음 (심사 탈락 위험)

**대응:**
- 로컬 데모 환경 사전 구성
- 녹화 백업 영상 (3개)
- 더미 데이터 사전 로드
- 30분 리허설 사전 진행

---

### Risk 4: 팀 멤버 병가/이탈
**발생 확률:** 낮음 (10%)  
**영향도:** 높음 (일정 지연)

**대응:**
- 각 기능별 2명 이상 숙지
- 문서화 충실 (코드 + 아키텍처)
- 핫라인 구성 (긴급 연락처)
- 백업 팀원 (외부 자문가)
